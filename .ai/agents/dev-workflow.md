# Plan dev-workflow - 开发工作流程

> **定位：** 记录项目的「批注循环 + 放手执行」开发方法论，包含完整的 ASCII 流程图。

---

## 一、核心原则

### 1.1 单向推进原则

- 严禁在缺少 Plan 文档的情况下直接开始写代码
- 先有想法，别上来就问
- 拿不准的时候，先问再动

### 1.2 隔离开发原则（针对老项目）

- 必须遵循模块化逐个击破
- 当前模块未通过最终测试前，绝不开启下一个模块的修改

## 三、详细流程说明

### 3.1 Step 1: 调研 Research

- **目标**：深入理解现有代码库
- **执行**：让 Agent 探索代码库结构
- **产出**：`plans/research/research-{功能名/版本号}.md` 文档
- **命名约定**：
  - 有对应开发 plan（有版本号）：`research-v{版本号}.md`
  - 只是想法/探索阶段：`research-{功能名}.md`
- **关键**：你先看完再往下

### 3.2 Step 2: 规划 Planning

- **目标**：制定详细的实施计划
- **执行**：让 Agent 输出完整计划
- **产出**：`plan-{功能名}-v{版本}.md` 文档，包含：
  - 代码片段
  - 文件路径
  - 技术权衡分析
  - UI 设计稿截图（`plans/screenshots/{模块名}/`）
- **模板**：`plans/template/plan_template_flutter.md`

### 3.3 Step 3: 批注循环（核心环节）

这步最精髓！直接在 plan.md 里觉得有问题的地方加批注：

| 批注类型 | 说明 |
|---------|------|
| 纠正错误假设 | 指出计划中不符合实际的地方 |
| 否决不合理方案 | 明确表示某个方案不可行 |
| 补充约束条件 | 添加技术或业务约束 |
| 画流程图解释 | 有疑惑直接让 Agent 画流程图解释并思考 |
| 标记 "don't implement yet" | 标记暂不实施的部分 |

**批注循环次数**：1-6 轮，直到计划完全没问题才放手！

### 3.4 Step 4: 放手执行

计划审批 OK 后，就一句话：

| 指令 | 说明 |
|------|------|
| `implement it all` | 把计划里的所有内容全部实现 |
| `mark completed` | 每完成一项就标记为已完成 |
| `don't stop` | 中间不要停，一口气做完 |

然后 Agent 就按计划一路执行完。

### 3.5 Step 5: 测试 + 修改

全部结束之后运行 `flutter analyze` 检查错误，根据需要继续做修改。

### 3.6 Step 6: Review 收尾

创建 `review-{功能名}-v{版本}.md` 文档，包含：

| 章节 | 内容 |
|------|------|
| 验收总览 | 功能完成度、质量指标 |
| Plan 偏差记录 | 计划与实际的偏差 |
| 遗留问题 | P0/P1/P2 分级管理 |
| 验收检查项 | 对照 plan 的检查点逐项验收 |
| 经验总结 | 做得好的、踩过的坑、下次改进 |

## 四、批注循环详解

### 4.1 批注类型

```
┌─ 批注类型 ────────────────────────────────────────────────────┐
│                                                               │
│  1. 纠正错误假设                                               │
│     例：「这里假设用户一定会登录，但实际有匿名访问场景」        │
│                                                               │
│  2. 否决不合理方案                                             │
│     例：「这个方案性能太差，改成批量操作」                      │
│                                                               │
│  3. 补充约束条件                                               │
│     例：「这里需要考虑并发安全，加锁」                          │
│                                                               │
│  4. 画流程图解释                                               │
│     例：「这块逻辑有点绕，画个流程图解释一下」                  │
│                                                               │
│  5. 标记暂不实施                                               │
│     例：「don't implement yet，先做核心功能」                   │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### 4.2 循环次数

| 轮次 | 说明 |
|------|------|
| 第 1 轮 | 发现明显问题 |
| 第 2 轮 | 深入细节，补充约束 |
| 第 3-4 轮 | 边缘情况、异常处理 |
| 第 5-6 轮 | 最终确认，确保无遗漏 |

## 五、Review 收尾详解

### 5.1 Review 文档结构

```
reviews/
├── review-{功能名}-v{版本}.md        # 功能收尾（必须）
├── code-review-{功能名}-v{版本}.md   # 代码审查（按需）
└── template/
    ├── review_template.md
    └── code_review_template_flutter.md
```

### 5.2 问题分级

| 级别 | 说明 | 处理方式 |
|------|------|---------|
| **P0** | 阻塞问题，功能无法正常使用 | 必须立即修复，才能签收 |
| **P1** | 重要问题，影响核心功能 | 必须关闭或转入新 plan 跟踪，才能签收 |
| **P2** | 次要问题，不影响签收 | 可遗留，但需在 review 中注明 |

### 5.3 签收条件

```
┌─ 签收条件 ────────────────────────────────────────────────────┐
│                                                               │
│  ✅ 所有 P0 问题已关闭                                         │
│  ✅ 所有 P1 问题已关闭或已转入新 plan 跟踪                      │
│  ⚠️ P2 问题可遗留，但需在 review 中注明                         │
│                                                               │
│  满足以上条件，Plan 才能标记为「已完成」                        │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

## 六、Plans & Reviews 系统

### 6.1 目录定位

```
┌─ plans/ ──────────────────────────────────────────────────────┐
│  职责：施工队                                                  │
│  - 规划做什么、怎么做、阶段验收                                │
│  - 计划文档、阶段跟踪、验收检查点                              │
│                                                               │
│  ├─ template/                                                 │
│  │   ├─ plan_template_flutter.md     # Flutter 主模板         │
│  │   └─ plan_prompt_template.md      # AI 提示词模板          │
│  │                                                           │
│  └─ research/                                                 │
│     职责：调研                                                 │
│     - 深入理解代码库、技术选型、可行性分析                     │
│     - research-v{版本号}.md（有开发 plan）                     │
│     - research-{功能名}.md（想法/探索阶段）                    │
│                                                               │
│  └─ screenshots/                                              │
│     职责：设计稿截图                                           │
│     - plans/screenshots/{模块名}/main_page.png                │
│     - plans/screenshots/{模块名}/detail_state.png             │
└───────────────────────────────────────────────────────────────┘

┌─ reviews/ ────────────────────────────────────────────────────┐
│  职责：审判官                                                  │
│  - 验收结果、记录问题、经验沉淀                                │
│  - 只做「发现和记录」，不做实施                                │
│                                                               │
│  └─ template/                                                 │
│      ├─ review_template.md             # 功能收尾模板          │
│      └─ code_review_template_flutter.md # 代码审查模板         │
└───────────────────────────────────────────────────────────────┘
```

### 6.2 目录结构总览

```
plans/
├── research/                          # 调研文档目录
│   └── ...
├── screenshots/                       # 设计稿截图
│   ├── hot_industry/
│   │   ├── main_page.png
│   │   ├── card_detail.png
│   │   └── header_decoration.png
│   └── industry_detail/
│       └── main_page.png
├── template/                          # 模板目录
│   ├── plan_template_flutter.md
│   ├── plan_prompt_template.md
│   └── README.md
├── plan-hot_industry-v1.0.md          # 带版本的计划
├── plan-industry_detail-v1.0.md
├── plan-toast-native-v1.0.md
├── plan-error-handler-improvement.md  # 独立功能（无版本）
├── plan-bottomsheet-flutter-fragment.md
└── README.md

reviews/
├── template/
│   ├── review_template.md
│   └── code_review_template_flutter.md
├── review-error-handler-improvement.md

└── README.md
```

### 6.3 Plan 命名约定

| 格式 | 示例 | 说明 |
|------|------|------|
| `plan-{功能名}-v{版本}.md` | `plan-hot_industry-v1.0.md` | 带版本的迭代开发 |
| `plan-{功能名}.md` | `plan-error-handler-improvement.md` | 独立功能/改造 |

### 6.4 Plan 状态生命周期

```
┌──────────┐    批注完成     ┌──────────┐    执行完成    ┌───────────┐
│  规划中   │ ──────────────► │  实施中   │ ────────────► │ review 中  │
└──────────┘                 └──────────┘               └─────┬─────┘
                                                             │
                                               ┌─────────────┴─────────────┐
                                               │                           │
                                      有 P0/P1 问题                P0/P1 已关闭
                                               ▼                           ▼
                                        ┌───────────┐               ┌──────────┐
                                        │ review 打回 │               │  已完成   │
                                        └─────┬─────┘               └──────────┘
                                              │                          ▲
                                     修复完成后重新 review ─────────────────┘
```

| 状态 | 含义 |
|------|------|
| 规划中 | plan 还在写，未开始实施 |
| 实施中 | 正在按阶段开发 |
| review 中 | 实施完成，等待或正在 review |
| review 打回 | review 发现 P0/P1 问题，需回去修复 |
| 已完成 | review 通过，所有 P0/P1 问题已关闭 |
| 已废弃 | 不做了 |

### 6.5 发现问题时的双向同步

```
reviewer 发现问题
    │
    ├─ 1. 在 review「遗留问题」表记录（主记录）
    │
    ├─ 2. 同步回 plan 阶段跟踪表
    │       阶段状态改为：⚠️ review 打回
    │       备注：review-{功能名} #问题编号
    │
    └─ 3. 修复完成后，双向更新
            review 遗留问题：⬜ → ✅ 已修复 YYYY-MM-DD
            plan 阶段状态：⚠️ review 打回 → ✅ 已完成（review 后修复）
```

**关键原则**：review 是问题的 source of truth，plan 只挂引用。

## 七、执行技巧

### 7.1 技巧 1: 反馈要短，指向已有代码

```
❌ 错误示范：
「这里需要实现一个用户登录功能，包括用户名密码验证、
  验证码、记住我、找回密码等功能...」

✅ 正确示范：
「这里参考 hot_industry 模块的写法，用同样的模式实现」
```

### 7.2 技巧 2: 方向错了？直接重来

```
❌ 错误示范：
「这块逻辑有问题，改成这样...然后这样...再那样...」

✅ 正确示范：
「方向错了，回到上个正确状态重新来过」
```

### 7.3 技巧 3: 结束后让 Agent 跑检查

```
全部结束之后：
「flutter analyze 检查错误」
→ 根据报告继续做修改
```
